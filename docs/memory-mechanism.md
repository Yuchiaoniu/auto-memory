# 對話記憶與壓縮搶救機制（白話說明）

這份文件用白話解釋我跟使用者的對話，怎麼在「自動壓縮」把細節抹掉之前先存起來，
壓縮之後又怎麼把這些細節念回給接手的我。裡面每個名詞都先解釋一次，不假設使用者已經懂。

這是「說明文件」，不是「規則」。每次對話都要遵守的規則放在 `CLAUDE.md`。

## 先搞懂三個背景名詞

**對話紀錄檔（transcript / .jsonl）**：
使用者跟我講的每一句話，軟體都會逐字存成一個檔案，副檔名是 `.jsonl`，
放在 `C:\Users\yuchi\.claude\projects\C--Users-yuchi\` 底下。這個檔案就是這次對話的逐字稿。

**自動壓縮（autocompact）**：
對話越聊越長，軟體要塞給模型的內容就越多。當內容累積到 15 萬 tokens
（在 `settings.json` 裡叫 `autoCompactWindow`），軟體會自動把前面一大段對話濃縮成一份短摘要，
再用這份摘要繼續聊。這個動作會丟掉很多細節，而且使用者無法控制它保留什麼。

**hook（觸發程式）**：
軟體在特定時機會去執行使用者指定的腳本。本機制用到兩個時機：
`Stop`（每一輪對話結束時）、`SessionStart`（一段對話重新開始時，含壓縮後接手）。

## 核心原則：規則全域、記憶各專案各自一份

- 「規則」全部專案共用一份，放在 `CLAUDE.md`，由使用者手動維護，搶救程式不會去動它。
- 「記憶」每個專案各自保存一份，放在各自資料夾的 `STATE.md`，由搶救程式自動維護。

也就是說，搶救程式只負責「這個專案目前做到哪、查到什麼」，不會去寫全域的 `MEMORY.md`。

## 目前實際在跑的兩支腳本

### 第一支：壓縮前自動存（寫入端）

腳本是 `C:\Users\yuchi\.claude\pre-compact-memory-save.ps1`，掛在 `Stop` 事件，
也就是每一輪對話結束都跑一次，設成 async（背景跑、不卡使用者）。它做的事如下：

1. 程式從 hook 的輸入取得目前的專案資料夾（cwd）。若這個資料夾沒有 `tasks.md`，
   程式判定這裡不是專案，直接結束，連 Haiku 都不呼叫，省 API 費用。
2. 程式打開最新的對話逐字稿，比對這次對話比上次存檔時多長了多少。
   成長沒超過 2 萬 tokens 就跳過，目的是省 API 費用；新對話會自動把基準歸零。
3. 程式抓最近 40 句對話，使用者的話留前 800 字、我的話留前 1500 字。
4. 程式把這些對話連同「這個專案既有的 `STATE.md`」一起送給便宜的 Haiku 模型。
5. Haiku 把值得長期保存的具體事實增修進 STATE.md，輸出更新後的完整 STATE.md；
   若這輪沒有新東西值得記，Haiku 回 `NO_UPDATE`，程式就收工。
6. 程式把結果寫進這個專案資料夾的 `STATE.md`。

Haiku 只記「具體事實與資料」：完成了哪個功能、建立或修改了哪支腳本/檔案、
文章寫了哪些段落、重要查詢結果（IP、地址、餘額、設定值、指令輸出、數字結論等）。
程式不記來回閒聊、語氣與過程性討論。

### 第二支：壓縮後自動念回（讀取端）

腳本是 `C:\Users\yuchi\.claude\session-start-inject-state.ps1`，掛在 `SessionStart`
事件、`matcher = compact`（也就是自動壓縮完成、我重新接手的那一刻）。它做的事如下：

1. 程式從 hook 的輸入取得目前的專案資料夾（cwd）。
2. 程式讀取該資料夾的 `STATE.md`（目前狀態與重要查詢資料）與 `tasks.md`（任務清單）。
3. 程式把這兩份內容印到 stdout，軟體會把這段內容注入「壓縮後接手的我」的 context。
4. 若兩個檔都不存在，程式判定這裡不是專案，靜默結束。

效果是：壓縮之後接手的我，一開場就讀得到這個專案的最新狀態與待辦，不必重新摸索、
也不會反覆詢問使用者已經講過的資訊。

## 一個重要修正：中文亂碼與檔案編碼

Windows 內建的 PowerShell 5.1 對 UTF-8 處理有兩個地雷，本機制都已避開：

- 呼叫 Anthropic API 時，舊寫法 `Invoke-RestMethod` 會誤判回應編碼，導致中文變亂碼。
  程式改用 `Invoke-WebRequest -UseBasicParsing`，再自行用
  `[System.Text.Encoding]::UTF8.GetString(...)` 把回應位元組按 UTF-8 解碼。
- 程式寫 `STATE.md` 時用「UTF-8 帶 BOM」，且所有 `.ps1` 腳本本身也存成「UTF-8 帶 BOM」，
  否則 PS 5.1 會把腳本裡的中文當成 Big5 解讀而弄壞。

## 與舊版的差異（給對照用）

最初版本是單一支 `post-compact-memory-update.ps1`，掛在 `PostCompact`，
而且只寫全域 `MEMORY.md`。後來改成現在的兩支腳本（壓縮前存、壓縮後念回），
記憶改成各專案各自一份 `STATE.md`，並移除了 PostCompact hook，避免每輪重複呼叫 Haiku。

## Context

此工具針對心理操控與情感暴力的識別與應對，目標使用者涵蓋施暴者、受害者、旁觀者三種角色。現有心理健康資源多為長篇文字或需專業背景，缺乏可快速查閱、以路徑為單位的互動工具。

技術環境：純靜態部署（GitHub Pages），無需後端服務，資料以 JSON 維護，確保社群可輕易貢獻與翻譯。

## Goals / Non-Goals

**Goals:**
- 提供完整的攻擊性路徑與正錨路徑索引，每條路徑可獨立查閱
- 三種使用者視角切換，內容與建議根據視角調整
- 旁觀者決策樹：依情境給出最省力介入建議
- 微積分框架視覺化：讓抽象的心理狀態變化可被直觀理解
- 完全靜態，可 fork 後自行部署

**Non-Goals:**
- 不提供個人諮商或診斷功能
- 不收集任何使用者資料
- 不替代專業心理治療
- 不建立社群功能或帳號系統

## Decisions

### 決策一：純靜態架構 vs 後端服務

選擇：純靜態（HTML + CSS + JS + JSON）

理由：
- 降低維護成本，任何人可 fork 部署
- 無隱私疑慮，資料不離開使用者裝置
- GitHub Pages 免費托管，降低社會成本

替代方案：Next.js / Nuxt SSR — 功能更強但增加部署複雜度，不符合最小阻力原則

---

### 決策二：框架選擇

選擇：Vue 3（Composition API）+ Vite

理由：
- 輕量、學習曲線低，社群貢獻門檻低
- Vite 建構速度快，靜態輸出乾淨
- 無需 TypeScript 強制，降低貢獻門檻

替代方案：原生 JS — 可行但路由與狀態管理複雜度高；React — 生態較重

---

### 決策三：資料結構

選擇：靜態 JSON 檔案，每條路徑為獨立物件

```json
{
  "id": "learned-helplessness",
  "name_zh": "習得性無助",
  "name_en": "Learned Helplessness",
  "type": "attack",
  "mechanism": "...",
  "signals": [...],
  "counter": [...],
  "user_lens": {
    "perpetrator": "...",
    "victim": "...",
    "bystander": "..."
  }
}
```

理由：易於社群貢獻，不需資料庫知識，可直接用 PR 新增路徑

---

### 決策四：視覺化框架

選擇：D3.js 用於微積分視覺化，CSS 動畫用於路徑地圖

理由：D3 在靜態環境下輸出 SVG，無需後端，且視覺表達力強

## Risks / Trade-offs

- [內容準確性風險] 心理學術語的中文翻譯可能有爭議 → 每條路徑附英文原名與來源引用
- [過度簡化風險] 將複雜心理路徑簡化為工具可能失真 → 強調工具為認知起點，非診斷依據
- [誤用風險] 攻擊性路徑資訊可能被用於操控他人 → 設計上攻擊性路徑的呈現以「識別」為框架，而非「使用」
- [維護風險] 純靜態架構在內容增長後管理複雜度提升 → 早期建立清晰的 JSON schema 與貢獻指南

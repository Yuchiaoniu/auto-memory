## Context

兩張查表是台灣統計學課程的核心工具，但傳統紙本查表麻煩且易出錯。互動工具可以：
- 任意 z 值精確計算（傳統表只到 2 位小數）
- 視覺化標出查到的陰影區域，強化「機率 = 面積」直覺
- t 表處理多自由度時不用翻多頁

## Goals / Non-Goals

**Goals:**
- Z 表工具：輸入任意 z（-4 到 4），即時顯示對應機率與曲線陰影
- t 表工具：輸入 df 與 α，回傳臨界值
- 兩工具都支援多種「目標機率類型」（左尾、右尾、雙尾、中段、半邊）
- 工具放在 glossary（速查頁）下方，跟既有資料並列

**Non-Goals:**
- 不支援卡方分配 / F 分配查表（範圍外）
- 不做歷史紀錄、不存使用者查詢
- 不取代靜態 Z 表與 t 表圖片（兩者並存：原 PDF 截圖供熟悉表結構、互動工具供實際解題）

## Decisions

### 決策 1：Z 表用即時計算（erf 函數），不用 hardcoded 表

**選擇**：用 JavaScript 內建 `Math.exp` + Abramowitz-Stegun 近似的 erf 函數即時算 $P(0 \le Z \le z) = \frac{1}{2}\text{erf}(z/\sqrt{2})$。

**理由**：
- 計算速度遠快於查表
- 精度可達 7 位小數，比紙本表（5 位）更精準
- 不限於 z 表預存的步進值（紙本 0.01 步進，互動可任意值）
- 與 area-viz.js 共用同一個 erf 實作

### 決策 2：t 表用 hardcoded 表 + 線性插補

**選擇**：把 PDF 附表 2 的數值 hardcode 進 JS 對象（37 列 × 8 欄 ≈ 296 個值），未列出的 df 用線性插補。

**理由**：
- t 分配 PDF 含 Gamma 函數，純 JS 即時計算精度不易控制
- 表只有 296 個數字，總大小不到 5KB，加進 JS 沒問題
- 學生實際用到的 df 大多是教科書列的值（1-30, 40, 50, 60, 80, 120）

**替代方案**：
- 即時計算用 Newton 迭代反查 incomplete beta function — 精度與速度都不易
- 用 CDN 載入 jStat — 多一個依賴

### 決策 3：UI 採「輸入 + 即時更新」

**選擇**：使用者輸入 z 或 df/α，按鍵時即時更新顯示結果（不需按「查詢」按鈕）。

**理由**：
- 跟既有 area-viz、chart-panel 一致的 UX
- 鍵盤輸入很快，計算很快，沒理由不即時

### 決策 4：放在 glossary 頁下方，而非單獨頁面

**選擇**：在現有 glossary 頁加入 `<h2 id="lookup">查表工具</h2>` section。

**理由**：
- 使用者已知道「需要查資料就去速查頁」這個心智模型
- 多一個頁面入口會增加導覽複雜度
- 兩個工具加起來大概 300 行 HTML+JS，不至於讓 glossary 變太長

### 決策 5：z 範圍限制與精度

**選擇**：z ∈ [-4, 4]，最多 4 位小數輸入精度（如 1.9600）。

**理由**：
- z = 4 時 P(0 ≤ Z ≤ 4) ≈ 0.49997 已經非常接近 0.5，超出 4 對學習無實用價值
- 紙本表也是到 z = 3.0 或 3.5，互動稍微延伸到 4 即可
- 4 位小數精度足以覆蓋所有教材會出現的值（1.96, 2.576 等）

## Risks / Trade-offs

- **風險：t 表線性插補在尾部精度差** → 緩解：在 α ≤ 0.001 與大 df 結合的極端值給出警告，建議使用其他工具
- **風險：使用者誤以為工具值就是答案，不會手動查表** → 接受：工具的目的本來就是讓「會用表」這件事更直觀，並非取代學會表結構（PDF 截圖仍保留）
- **權衡：hardcode 表佔約 5KB JS** → 接受，比載外部庫好

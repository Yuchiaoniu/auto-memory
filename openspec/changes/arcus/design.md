## Context

系統現況：routing_law.py 同時承擔「規則定義」與「自我演化」兩個職責，每次演化需修改 Python 代碼。Kitchen 運行在 GCP VM（forest-carbon），server.py 透過 SSE 串流回應瀏覽器，kitchen_hooks.py 在每輪 prompt 建構前後執行。

目標：把演化載體從「代碼文字」轉為「數值向量」。

## Goals / Non-Goals

**Goals:**
- routing_law.py 凍結，不再因學習而改動
- 每條規則有獨立的 weight 值，weight 決定注入強度
- 每輪失敗事件自動記入 error_tensors.json
- weight_baker.py 可手動或定時對 weights 做梯度調整
- Kitchen 現有所有檔案納入 arcus 專案作為版控基準

**Non-Goals:**
- 不做神經網路或 embedding（保持可解釋性）
- 不做 A/B testing 框架
- 不改動 server.py 的 SSE 串流邏輯
- 不做跨 VM 的分散式 weight 同步

## Decisions

**D1：weight 存為 JSON，不存資料庫**
每輪啟動讀一個小 JSON（< 10KB）比連資料庫快，且人工可直接編輯做強制覆寫。

**D2：error_tensors.json 只追加，不覆寫**
失敗事件是訓練信號，不能刪，baker 跑完後只更新 DNA_weights.json，error_tensors.json 保留完整歷史。

**D3：routing_law.py 的 weight loader 在 import 時執行一次**
不做每輪重新讀取（避免 I/O），服務重啟時自動載入最新 weights。輕量調整（< 0.05 變化）不需要重啟。

**D4：weight 閾值設 0.3**
weight < 0.3 的規則不注入 prompt（省 token）；weight ≥ 0.3 依值線性縮放注入強度。初始值全設 0.5（中性）。

**D5：baker 使用簡單指數移動平均**
`new_weight = old_weight * 0.85 + hit_rate * 0.15`
不用 SGD（不需要 PyTorch），純 Python 計算，GCP VM 免 GPU。

## Architecture

```
每輪對話流程：
  輸入 prompt
    → kitchen_hooks.py before_prompt_hook()
      → routing_law.py run()
        ★ 讀 DNA_weights.json（已在 import 時載入記憶體）
        → 把 weight ≥ 0.3 的規則注入 prompt，強度依 weight 縮放
    → Claude CLI 呼叫
    → kitchen_hooks.py after_response_hook()
      → routing_law.py scan()
        ★ 偵測違規 → 序列化進 error_tensors.json（追加）

每日 / 每 5 次違規後（cron 或手動）：
  weight_baker.py
    → 讀 error_tensors.json
    → 計算各規則命中率與平均偏差 L
    → EMA 梯度調整
    → 覆寫 DNA_weights.json
    → 下次 routing_law.py import 時生效（需重啟）
```

## Risks / Trade-offs

- [weight 漂移] → baker 設 min/max clamp（0.1–0.95），防止規則被永久抑制或過度強化
- [error_tensors.json 無限增長] → baker 跑完後封存 > 30 天的記錄到 error_tensors_archive.json
- [routing_law.py 凍結後發現 bug] → 允許 hotfix commit，但每次 hotfix 需在 arcus tasks.md 開票記錄

## Hotfix 程序（routing_law.py 凍結後的 bug 修復流程）

routing_law.py 在 arcus 架構下被視為「凍結的物理定律」——只有 DNA_weights.json 應演化。
若發現 routing_law.py 本身有 bug，遵照以下流程：

1. **在 arcus/tasks.md 新增 [ ] 條目**
   格式：`- [ ] hotfix-YYYY-MM-DD：<bug 描述與修復方案>（HH:MM）`

2. **修復 routing_law.py**
   只改 bug，不擴充功能，不改任何數值常數（常數變更走 DNA_weights.json）

3. **更新 kitchen-snapshot/**
   `cp routing_law.py arcus/kitchen-snapshot/routing_law.py`

4. **把 tasks.md 條目標記為 [x]**
   記錄實際修改摘要（改了哪幾行、修了什麼）

5. **（選擇性）調整 DNA_weights.json**
   若 bug 是某條規則的 false positive 造成，考慮直接降低該規則 weight，
   而不是修改 regex —— 數字演化優先於代碼變更。

# 記憶卡片機制重構（2026-07-18）

## 為什麼要改

現行機制（`_memory_maybe_extract`）有三個落差，卡片累積了三個月只存下 16 張、
而且科別和使用者真正在跑的專案（統計學、林業碳匯、資訊管理）完全對不起來：

1. **每 4 輪才看一次**：`interval` 預設 4，四分之三的對話直接丟掉，不是「漏掉不重要的」，
   而是「用輪次編號隨機丟掉」。
2. **科別當場由模型決定，而且模型看不到既有抽屜**：每輪各判各的，抽屜只會發散不會收斂。
   `_subject_canonical` 只做尾綴吸收（學／論／理論／科），字根不同一律開新抽屜。
   注意：把寫入位置從索引搬到 memory.md **不會**修好這一點，問題出在判斷當下缺少全局視野。
3. **作品卡沒有任何生產者**：`_memory_extract` 永遠只用 `'term'` 呼叫 `remember`，
   七個抽屜的 `works` 全是空的。

另有兩個工程隱患：抽取跑在 `daemon=True` 執行緒上，主行程結束就被無聲砍掉；
`remember` 每存一張卡就整份重寫一次索引。

## 三段式設計

```
每輪（確定性收料）        每 6 小時（收斂歸檔）          回覆前（確定性注入）
─────────────────       ──────────────────────       ──────────────────
_memory_capture_turn    _memory_archive_daily        _memory_keyword_hits
  便宜模型抽名詞          一次讀進整批收件匣            用來訊息比對卡片標題
  科別標 provisional      連同現有抽屜清單給模型          取前 ~8 張注入 prompt
  不做正規化              「能併就併，不輕易開新抽屜」      完全不經模型判斷
  追加進 memory.md        合併同名卡，留較完整的解釋
  機器區塊（檔尾）         整份索引只寫一次
                         清空收件匣
```

關鍵切分：**Python 決定「何時、要不要做」，模型只決定「內容是什麼」**。
排程、去重、注入全部是確定性程式碼，不靠提示詞叫模型記得做——大型語言模型終究是機率生成器。

## 決策記錄

### D1. 排程放在 arcus_core.py，不放 server.py

`server.py` 是常駐行程（`import arcus_core`），`python3 arcus_core.py --mcp` 是每輪被
claude CLI 生出來、講完就死的臨時行程。歸檔要定時執行，直覺會想寫在常駐端。
但寫在 `server.py` 只有一行 `start_memory_scheduler()`，維護時很容易被刪掉而沒人發現。

改為：排程器寫在 `arcus_core.py`，模組載入時自行啟動，用一行
`if '--mcp' not in sys.argv` 自我檢查擋掉臨時行程。常駐端 import 就自動有排程，
臨時行程不受影響，沒有任何一行需要維護者記得加。

### D2. 每 6 小時一次，用輪詢，不用精準睡到指定時刻

```python
while True:
    _memory_check_and_archive()   # 距上次歸檔超過 6 小時 → 執行
    time.sleep(600)
```

輪詢是**無狀態**的：每次醒來都重新從系統時鐘減去索引裡的上次歸檔時間戳，推導一次要不要做，
系統調時間、休眠喚醒、行程重啟都會自動補做。精準睡眠帶著「我會在 T 準時醒來」的狀態假設，
一旦錯過那個窗口，那批就永遠跳過、沒有補救。間隔 10 分鐘，同時決定重啟後的補做延遲上限。

不用作業系統層的 cron：cron 起的是另一個行程，無法共用常駐記憶體裡的索引，
反而把並行歸檔的問題重新製造出來。

### D3. 歸檔必須互斥，重複歸檔不是無傷大雅

觸發條件是「距上次歸檔已超過 6 小時」這個持續成立的區間判斷，不是某一秒，
所以每個 6 小時窗口過後的第一個行程都會命中，一天命中四次，不是罕見狀況。
而且 `os.replace` 只保證檔案不會寫壞，不保證不掉資料——兩邊各自從自己記憶體裡的
舊快照整份重寫，後寫的會蓋掉先寫的成果。

用 `O_CREAT|O_EXCL` 開一個佔位檔當鎖，搶不到的直接跳過。鎖檔是歸檔期間的暫存物，
結束就刪。

### D4. 不產生一堆新檔案

| 要存的東西 | 放哪裡 | 是否新檔 |
|---|---|---|
| 收件匣（當天原始卡片） | 各專案 memory.md 檔尾的機器區塊 | 否 |
| 抽屜索引 | 既有 memory_index.json | 否 |
| 輪次計數 | 既有 .smart_brake_state.json | 否 |
| 上次歸檔時間戳 | memory_index.json 內底線開頭的保留鍵 | 否 |
| 歸檔鎖 | 暫存佔位檔，用完即刪 | 過渡性 |

保留鍵放進索引後，`list_subjects` 與 `recall` 必須把底線開頭的鍵濾掉，否則會被當成抽屜列出來。

### D5. 作品卡接到既有指令上，不另做觸發

- `cmd=draw`：標題、網址、路徑當下都齊全，直接開卡。
- `cmd=promote`：標題取落地檔名、路徑取落地位置，摘要留空，等當天歸檔時補。

## 風險與取捨

- [每輪都呼叫便宜模型，成本上升] → 保留 `_memory_cheap_gate` 長度門檻擋掉寒暄短句；
  收料階段只做抽取不做判斷，用最便宜的模型。
- [收件匣讓 memory.md 變長] → 機器區塊固定在檔尾且每 6 小時清空，長度上限是「6 小時的量」；
  歸檔後 memory.md 反而比現在短。
- [歸檔當下模型仍可能誤判科別] → 這次它看得到現有抽屜清單與整批卡片，
  比每輪盲判有全局視野；判錯的抽屜可事後合併，資料不會掉。
- [STATE.md 歸檔同時上車會放大爆炸半徑] → 這次不做，列為第二階段。
  搬移時必須「只搬不刪」：先追加進 memory.md 並確認寫入成功，再從 STATE.md 移除，
  移除前先留一份帶日期的備份。


# 論文卡片機制（2026-07-19）

## 要解決什麼

固定的閱讀模式是：讀一篇論文，講出研究問題、研究方法、研究流程、研究成果、
爭議、未來發展。這六欄每篇都要，而且同一篇之後一定會再看。

浪費有兩處。全文讀完之後整份壓在對話裡，後續每一輪跟著重算。
以及換一個對話再問同一篇，又從頭讀一次，而 arcus 一律用 Opus，重讀特別貴。

## 三段流程

    收到來源 ──▶ 算內容雜湊 ──▶ 命中快取？──是──▶ 直接回卡片，結束
                                    │否
                                    ▼
                            抽文字（PyMuPDF）
                                    ▼
                        切段合併至約 1000 字元，編鍵 p012#03
                                    ▼
                        組出「每段掛鍵」的全文當提示詞
                                    ▼
                    開一次性模型行程（Opus，上限 600 秒）
                                    ▼
                    程式回查出處鍵是否真實存在 ──▶ 假的標記出處存疑
                                    ▼
                        存 meta.json / card.json / text.json
                                    ▼
                        對話只收到卡片，全文從未進過對話

## 決策

**D1：不靠子代理，靠一次性模型行程。**
arcus 沒有子代理機制。隔離改用 arcus_core 既有的 _call_model，
它開的是獨立子行程 claude --print，該行程有自己全新的內容空間，
跑完就結束，對話這一側看不到那四萬字元。效果等同隔離。

**D2：卡片與全文拆成兩個檔，不合併。**
卡片幾乎每次都讀，全文幾乎不讀，存取頻率差兩個數量級。
放同一個檔會逼著每次都碰到大檔。

**D3：無損靠定義達成，不靠事後檢查。**
整份文字直接定義為「各段用固定接合字串串起來」，還原就不可能出錯。
雜湊值只是最後一道保險，不是主要手段。

**D4：出處鍵由程式驗證，不靠模型自律。**
模型回傳的鍵拿去切段全文裡回查，查不到就標記出處存疑。
與記憶卡片同一原則：程式判斷有沒有，模型只負責內容是什麼。

**D5：去重看檔案內容，不看檔名。**
改名、換資料夾、重複下載都會命中同一份快取，不重跑也不重複付費。

**D6：抽取套件延後匯入。**
臨時 MCP 行程每輪都載入 arcus_core，不該為了論文功能付抽取套件的啟動成本，
所以 import paper_cards 寫在指令處理區塊裡面。

## 實測結果（2026-07-19，資料由虛擬機自行下載）

測試材料：arXiv 1706.03762v7 Attention Is All You Need，15 頁。

- 抽出 539 個碎塊，合併成 34 段，平均每段 1177 字元，全文 40,035 字元
- 接回去與原文一字不差，存檔後重算雜湊值同樣相符
- 萃取六欄耗時 57.3 秒，出處存疑欄位 0
- 卡片 2,139 字元，相對全文 40,035 字元，省下約 95%
- 第二次收同一篇命中快取，耗時 0.2 秒
- 跨論文只取研究方法一欄，回傳 474 字元
- 驗證 18 項全數通過

## 尚未支援

超過 15 萬字元的專書要分批讀再合併，目前直接回報不支援。
掃描型檔案抽不到文字時回報需要文字辨識，不硬產半空的卡。


# §87 廢除 STATE.md，待辦封存併入記憶卡片的歸檔時鐘（2026-07-19）

## 要解決什麼

現況快照檔（STATE.md）原本設計成「整段覆寫、只留一段現況」，結構上不該膨脹。
實際上全部 29 份合計 132,787 位元組，最大的一份 2,286 行，
原因在於覆寫工具（write_state.py）的預設路徑寫死在單一專案，
其他專案沒有任何程式在覆寫，只有對話端不斷手工追加，於是一路長大。

更關鍵的是，§78 打底瘦身之後，伺服端已經不再每輪注入現況快照檔。
它已經沒有讀者，只剩專案清單畫面拿去顯示，等於是一份沒人看的檔還在長大。

## 決策

**D1：現況快照檔整個廢除，不做封存機制。**
封存只是把膨脹速度放慢，沒有解決「三個檔語意重疊」的根因。
專案狀態改由兩個檔承擔：待辦清單（tasks.md）記進度，記憶檔（memory.md）記查詢資料與姿態。

**D2：只搬不刪。**
29 份先整份備份到 backup/state_retire_20260719，
再逐份附加進各專案的記憶檔，程式回查特徵字串確認寫入成功，才准刪原檔。

**D3：待辦封存只搬已勾選，不搬未勾選。**
既有的 log_task.py archive 屬於「隨用即刪」，整區搬走且含未勾項目，
適合人工收尾，不適合定時自動跑——定時跑會把 441 條待辦一次清空。
所以另寫只搬已勾項目的版本，未勾項目原地留著。

**D4：兩個歸檔共用同一個時鐘，不另外開排程。**
排程迴圈先問一次「該歸檔了嗎」，記憶卡片歸檔會蓋掉時間戳，
所以必須先把答案存起來再依序做兩件事，否則待辦封存永遠輪不到。

**D5：併發沿用既有那把鎖。**
待辦檔用 flock 排他鎖，和 log_task.py 的打勾、新增章節是同一把，兩邊同時動不會互相蓋掉。

## 改了哪些地方

- server.py：專案清單不再讀現況快照檔，欄位保留空字串供舊前端相容
- status.sh：改看待辦封存檔，不再看現況快照檔
- arcus_core.py：新增待辦封存三個函式，接進記憶卡片的排程迴圈
- write_state.py：整支退役，改名保留
- close_round.py：現況快照那一步改成印一行說明就跳過，不碰任何檔案

## §88 讀可以跨專案，寫不可以（2026-07-19）

D1 原本 _arcus_safe_path 同時管讀與寫，兩者共用同一道柵欄，放寬讀就等於放寬寫。
   改法是把讀取拆出獨立函式 _arcus_read_path，寫入完全不動。

### 問題定位

實測數據：對照組極短提示耗時 6.2 秒，正式組真實提示（8965 字元）耗時 302.1 秒、
回應 7916 字元、解析出 99 則名詞。模型本身完全正常，設定的 180 秒逾時低於實際所需，
每次都在快要完成時被切斷。原始碼裡各處逾時值（120、90、180）彼此不一致，
可見 180 秒是隨手填的數字，不是量測出來的。

### 決策

**D1：讀出行程編號後檢查存活，不只看陳舊秒數。**
舊版只比對修改時間是否超過門檻，持有者當掉後仍要空等。新版先問行程是否還在，
已死立刻回收。只攔 `ProcessLookupError`，不攔全部 `OSError`——
`PermissionError` 代表行程存在但屬於別的使用者，此時絕不可回收。

**D2：把慢的動作移出鎖的範圍。**
鎖的用途是保護索引檔的讀改寫，不是保護模型呼叫。三段式後鎖只在收料與併入時各持有一次，
兩次都在 1 秒內完成。

```
收件匣（各專案 memory.md 內的機器區塊）
      │  短鎖：取走並清空
      ▼
成品區 _memory_spool/raw_*.json      ← 卡片在此落地即安全
      │  不佔鎖：模型分批收斂，每批 50 張
      ▼
成品區 stage=ready
      │  短鎖：併入索引，成功才刪檔
      ▼
memory_index.json
```

**D3：卡片先落成品區才清空收件匣。**
任一段中途當掉，卡片都留在成品區，下一輪從 `stage` 欄位接續，不會回到起點也不會消失。
模型失敗時檔案維持 `stage=raw`，下一輪重試整份。

**D4：分批上限 50 張、逾時 600 秒。**
舊版用 `terms[:400]` 一次送出並直接丟棄超出的部分，屬於無聲截斷。
分批後全部卡片都會被處理，單批提示長度也降到模型能在逾時內回完的規模。

**D5：收料時補上來源專案名稱。**
成品區把所有專案的卡片放在同一份檔案，靠每張卡片自己攜帶 `project` 欄位區隔來源，
不需要為每個專案各開一份檔案。

**D6：沿用「後定義覆蓋前定義」的增修慣例。**
新版 `_memory_archive_batch` 與 `_memory_lock_acquire` 定義在原版之後，
模組載入時後者覆蓋前者。原版留在檔案裡作為對照，不刪除。

### 驗證

假模型測試 20 項全通過，涵蓋分批數量、成品區清空、模型失效時卡片保全、
下一輪補回、以及鎖對已死／存活／無鎖三種情況的判斷。
真實資料實跑取走 35 張，索引名詞由 198 增至 217，收件匣與成品區皆歸零。
外部網址 `https://forest-carbon.duckdns.org/arcus/` 回 200。

## §90 工具合併設計 (2026-07-19)

### 決策

**D1：合併範圍以「引擎實際用到的」為準，不是把 tools 整個搬進來。**
清點後只有三支被呼叫，合計 2205 行，併入後核心約 5900 行。
其餘工具與引擎無關，不納入。

**D2：第三方套件改成容錯匯入，不讓單一缺件拖垮整個引擎啟動。**
上網搜尋那段原本在模組層直接匯入 requests 與 bs4。若照原樣併入，
缺套件時整個引擎會在載入階段就死掉，等於把一項功能的故障升級成全面故障。
改為包在容錯匯入裡，缺套件時只有該項功能失效並印出訊息。

**D3：畫圖維持子行程呼叫，但引擎指向自己。**
畫圖有大量渲染與字型狀態，放進常駐行程會累積記憶體。
保留子行程，只把引擎路徑常數改指向 arcus_core.py 自己，
並在命令列進入點加一個 --spec 分支。子行程不啟動歸檔排程，避免互相搶鎖。

**D4：命令列進入點必須排在檔尾。**
併入的函式排在進入點之後時，以腳本方式執行會找不到函式（實際踩過，NameError）。
進入點移到所有定義之後即解決。

**D5：改名只改函式定義與呼叫處，不做全字詞替換。**
第一次合併時用全字詞比對把 main 改名，結果連 Word 文件命名空間網址結尾的 main
也一起被改（網址是 .../wordprocessingml/2006/main），文件結構對不上、產圖失敗。
從備份還原後改為只替換定義與呼叫，問題消失。這個教訓值得記住：
對整份原始碼做識別字替換時，字串內容也會被掃到。

**D6：能力必須同時出現在三個地方才算揭露。**
分派點接上、工具清單的 enum 列出、姿態檔寫明——缺任何一個，模型就用不到。
論文抽取五個指令正是只做了第一項，於是引擎有能力卻不知道自己有，
對使用者回答「我讀不到 pdf」。這不是模型說謊，是說明書漏寫。

### 驗證

同行程實測五項全過（畫圖產圖回傳網址、讀 pdf 建卡成功、發佈網頁、上網抓文 127 字、論文清單 1 筆）。

## §91 論文卡片改造：權威索引庫連接器 + meta 逐字化 (2026-07-20)

科別走逐篇分類（乙），來源改用權威索引庫的官方介面，不爬期刊 HTML、不用 pdf 內部中繼資料。


**D3：不使用中繼資料，指的是 pdf 檔案內部的中繼資料（fitz doc.metadata）。**
索引庫的權威著錄是人工策展、可信，正是作者與科別的逐字來源，兩者天差地別。

## §92 設計思考原則學習機制：條件式原則取代舊憲法法條權重 (2026-07-22)

### 要解決什麼

使用者希望 arcus 能判斷他每次實質回答背後的本質，揣摩他的設計思考模式，
省去一再重複解釋同一套想法的時間。舊的「憲法法條權重自適應」機制因為過於笨重、
且有邏輯謬誤而被棄用，這一版重新構想，從架構上避開舊機制的三個致命缺陷。

### 舊機制的三個致命缺陷（棄用原因）

舊機制的比喻是「程式碼＝永不改變的物理定律，DNA_weights.json＝靠反向傳播持續演化的質量」。
三個架構層缺陷：

1. **反向傳播沒有真標籤**：反向傳播需要正確答案當梯度，但「這次路由對不對」根本沒有標準答案，
   於是拿代理訊號（出現頻率當正確性）當梯度，形成頻率越高越被強化的自我循環，越跑越偏。
2. **純量壓掉條件**：一條規則只給一個浮點數，無法表達「在情境 C 成立時才適用」，
   高權重規則因此在錯誤情境也被套用。
3. **質量持續演化＝漂移又難稽核**：浮點矩陣不斷漂移，人看不懂，無法稽核。

前兩點對應使用者說的「邏輯謬誤」，第三點對應「笨重」。

### 新機制設計

引擎用自己的語言判斷，把使用者答案的本質萃取成一條白話條件式原則，存成可讀 jsonl。
信心用離散計數表示，不做梯度、不存浮點權重，從構造上對三個缺陷免疫。

- 原則格式：`當 {情境when} → 偏好 {then}（本質 because，適用範圍 scope）`
- 信心：observation 觀察數／contradicted 被打臉數，非浮點權重。
- 升級門檻：觀察數達 2 且未被打臉 → 由暫定（tentative）升為上線（live）。
- 更新訊號：靠使用者明確更正或認可，不靠出現頻率。
- 衝突解決：靠 scope 的具體程度，不靠數值大小。
- 跨專案共用：使用者設計思考模式不分專案，存於全域單檔 `arcus_user_model.jsonl`。
- 派工指令：principle_add 收料、principle_hit 固化與打臉、principle_list 檢視。
- 每輪注入：上線原則餵進系統提示，命中即照做並報備，被更正即退回暫定。

### 決策記錄

#### D1. 條件式原則取代浮點權重

一條原則明確標出 when/then/because/scope，人一眼可讀、可稽核。這同時解掉「純量壓掉條件」
與「難稽核」兩個缺陷。

#### D2. Python 當笨儲存，引擎做萃取與比對

萃取、比對、判斷全交給引擎當作判斷力（透過系統提示教學），Python 只負責讀寫檔。
這貼合 arcus 每輪開新子行程的無狀態架構，也徹底繞開反向傳播的陷阱——沒有梯度，就沒有代理訊號自我循環。

#### D3. 離散計數取代梯度，靠明確訊號更新

觀察數與被打臉數都是整數計數，升級與退回都由明確的認可／更正觸發，不由頻率驅動，
根除「頻率當正確性」的自我強化迴圈。

#### D4. 區分過程型與結論型原則，並立「預設值而非眼罩」規則（2026-07-22 決策）

**問題**：原則若無差別地一律「命中就照做」，會壓縮引擎的創造性。因為注入開場一句
「依你原則我選了X」會產生錨定效應，把答案框住，沒走的另一條路拿不到版面。

**區分兩類原則**：

- **過程型／風格型**（例：面對不可逆外推又有分歧時先攤清狀況再問範圍；回覆先講結論再展開）：
  只規範引擎切入問題的姿態，不碰答案內容，自動套用純粹省時，創造性零損失。
- **結論型／內容型**（例：架構一律選最簡方案）：直接鎖定答案內容，無差別自動套用會壓掉發散。

**決策——原則是預設值，不是眼罩**：結論型原則命中時，引擎照樣以該偏好當主推，
但只要這一次真有一個實質更好或更有意思的替代方案，就必須點出來，例如
「依你〔id〕原則我選了X；不過這次Y可能更合適／更值得一看，因為……」。
主推維持省時，road-not-taken 也保住一口氣；學到的偏好只設定起點、永遠不封路。

### 驗證結果（2026-07-22）

部署驗證三關全過：服務 active、外部網址回 200、三個新指令實際能寫能讀。
功能測試（走暫用檔、跑完即刪，不碰正式全域檔）逐項通過：新增暫定原則、
相同原則去重並升上線、上線後注入區塊帶出、打臉退回暫定、打臉後注入不再帶出、列出正常。

### 落地狀態（2026-07-22 更新）

D4 的「預設值而非眼罩」規則與過程型／結論型的區分，已於 2026-07-22 落到系統提示的教學段（commit 9501283），引擎每輪讀到的指示現已實際照這條規則運作，並經重啟＋外部網址回 200 驗證。


## §93 語意優化雙層機制：機械層直改＋判斷層自檢，共用原則學習骨架 (2026-07-23)

### 要解決什麼

使用者寫文章（尤其論文）時，反覆需要同一組語言修整：把「本研究」換成「此研究」、把話留有餘地而不講死、專有名詞或代名詞先給白話再帶出。這些修整若每次都口頭交代，很花時間。此機制要讓引擎把這組語言偏好學起來、日後自動套用，並且和 §92 的設計思考原則機制共用同一套學習骨架，不另造第二套輪子。

這次先談清楚邊界：跟使用者對話的語氣不列入規範，使用者明確表示對話「自由發揮就好」。機制真正服務的場景是寫文章。因此機制帶一個「模式感知」的設計——在對話模式下走精簡、每點簡短；在文章模式下改走「完整但不冗長」。同一套規則依場景切換鬆緊，不是一套標準壓到底。

### 為什麼要分兩層

語言修整有兩種性質，硬綁在一起會兩頭落空：

1. 一類修整的規則明確、可窮舉、替換安全，例如「本研究→此研究」。這類交給程式碼做確定性替換，又快又不漏，不必動用語言模型判斷。
2. 另一類要讀懂語意才知道該不該動，例如「這句話有沒有講太死」「這個專有名詞前面有沒有先給白話」。這類無法用字典窮舉，必須靠語言模型當場判斷。

硬用程式碼去猜第二類，會誤傷該保留的句子（把合理的肯定句也改軟）；硬用語言模型去做第一類，則又慢又不穩（同一個替換有時做、有時忘）。所以分兩層：機械層負責確定性的部分，判斷層負責需要理解語意的部分。

### 決策記錄

#### D1. 機械層：確定性字典加正則，只做保守替換與標記

機械層是一段程式碼，做兩件事。第一，對「安全替換清單」裡的詞直接改，例如把「本研究」換成「此研究」、把「本專案」換成「此專案」，這種替換沒有語意風險，改了就對。第二，對「絕對詞清單」（證明、必然、一定、絕對、完全、永遠、所有、毫無）只做標記、不自動改，因為一個絕對詞該不該留要看上下文，交給判斷層決定。機械層守一條保守原則：拿不準的一律不改，只標記，把決定權讓給判斷層。

#### D2. 判斷層：系統提示裡的寫作自檢，處理語境與需判斷的語法

判斷層不是程式碼，而是寫進系統提示的一段自檢指示。引擎在產出文章段落前，依這段指示檢查四件事：長度落在「完整但不冗長」的區間、把話講得留有餘地、專有名詞與代名詞先給白話再用、被問「怎麼做」時先分析再收斂到做法。這些都要讀懂語意才能判斷，無法用字典代勞。

#### D3. 共用 §92 的學習骨架，不另造機制

語言偏好的學習方式，和 §92 的設計思考原則完全同構：一條規則存成可讀的條件式（當某情境 → 偏好某寫法，本質是某原因），信心用離散計數表示（觀察數／被打臉數），不做梯度、不存浮點權重，靠使用者明確認可或更正來升級與退回。差別只在儲存的內容換成語言偏好，機制骨架照搬，不重造。

#### D4. 文章可當萃取來源，不可當梯度訓練語料

使用者問過能不能拿文章來訓練這套機制。可以，但方式要說清楚：引擎讀一篇使用者認可的文章，從中萃取出幾條可讀的語言偏好規則，交給使用者確認後才存起來。這是「讀文章、萃取白話規則、人工確認」，不是把文章當語料去調一個浮點矩陣。後者正是舊機制「拿頻率當正確性」的老陷阱，必須避開。

#### D5. 後設分析姿態納入判斷層（2026-07-23 決策）

使用者要求寫作與分析時採取後設姿態：被問「某件事該怎麼做」時，不直接跳到步驟，而是先分析問題的結構、假設與取捨，再收斂到做法。這條和 §92 的設計思考模式同源（先把狀況攤清楚，再收斂），在寫作與分析場景一體適用，故納入判斷層當一條寫作自檢規則。

#### D6. 兩種自適應規則寫進同一支模組，觸發閘門切斷注入膨脹，先獨立後併入 core（2026-07-23 決策）

**問題**：把學到的規則全部注入系統提示，會隨學習越來越龐大，膨脹到後來每條都被稀釋、反而全都跟不好。注入除了軟（模型可能忘），還有無限膨脹這第二個毛病。

**決策一，統一模組**：思考原則與語言偏好兩種自適應規則，寫進同一支模組 arcus_adaptive.py，共用同一套學習骨架（收料／固化／打臉／升級），語言偏好那邊再多一段機械掃描。兩份儲存檔分開（arcus_user_model.jsonl 與 arcus_style_model.jsonl），但機制程式碼合一，不重造。

**決策二，觸發閘門**：每條規則存的時候帶一個觸發條件（關鍵字加模式，模式分對話／文章）。一支確定性的比對腳本，拿這一輪的情境去比對觸發條件，只注入命中的少數幾條。注入量從此跟「這一輪相關的幾條」走，不跟「總共學了幾條」走，膨脹就此切斷。這一招沿用使用者全域規則檔既有的觸發式參考（提到某關鍵字才讀某參考檔），把它套到學到的規則上。

**決策三，先獨立後併入**：先做成獨立模組、測到用得順，再併進 core，讓 arcus_core.py 匯入它。

**驗證（2026-07-23）**：獨立模組已建成，通過 15 項功能測試——學習骨架的新增／去重升級／打臉退回、觸發閘門的對話不注入／文章命中關鍵字才注入／文章無關鍵字不注入、兩份儲存檔互不干擾、機械掃描的安全替換與絕對詞只標記不改。尚未併入 core，待使用者授權。

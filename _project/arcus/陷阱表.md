# Arcus 陷阱表

> 記錄踩過的坑與根治教訓，避免重蹈。

## 坑 1：刪資料夾造成 Claude Code 自鎖（2026-07-13）

**現象**
bootnode 上的 host 端 Claude agent 每次呼叫任何工具（Read／Grep／Glob／Edit／Bash／Write）
都直接中止，回報「No such file or directory」，整個 agent 動彈不得，形同自鎖。

**根本原因**
`~/.claude/settings.json` 的 `PreToolUse` 掛鉤，`command` 指向舊路徑
`tools/project-arcus/workflow_gate_hook.py`。當那個 `tools/project-arcus/` 資料夾被刪除後，
掛鉤腳本隨之消失。PreToolUse 會在每個受 matcher 命中的工具「執行前」先跑掛鉤，
掛鉤檔不存在就讓該次工具呼叫直接失敗、中止——於是每個工具都被擋，形成自鎖。

**為何 SSH 能救**
本機透過非互動式 SSH 在遠端直接跑 bash，不經過 host 端 agent 的 settings.json 掛鉤，
所以外部能在 agent 凍結時修好設定。

**根治**
把 `settings.json` 的 PreToolUse 錨點改指向正規常駐位置
`openspec/changes/arcus/runtime/workflow_gate_hook.py`（已備份 `settings.json.bak.arcusfix`），
並以模擬 PreToolUse payload 驗證掛鉤 exit 0。這樣就算舊路徑那份複製檔日後再被清掉也不會自鎖。

**教訓（最高優先）**
**刪任何資料夾前，必須先全域搜 `settings.json` 有沒有掛鉤路徑指向該目標。**
掛鉤指向的檔一旦連同資料夾被刪，會讓整個 agent 自鎖，且只能從外部（SSH／改設定）解。

## 坑 2：掏空功能群卻留下宣稱與注入殘留（2026-07-13）

**現象**
`arcus_core.py` 標頭宣稱「六支合體」，但第三群（反向傳播 bake）、第四群（法條層 run／scan）
早被掏空成「註解標頭＋大段空白行」，無任何實作。更嚴重的是 `_RECORD_WRITER_MAP` 與
`ARCUS_ARCHITECTURE` 兩段字串**每輪都注入系統提示**，內容列出 `bake()`／`scan()`／
`append_action_trace`／`write_checkpoint`／`_append_court_case` 等不存在的函式，
等於每回合對執行中的 agent 謊報它有這些能力。

**根治**
外科刪除：移除空殼區塊 3／4（99 行）、刪掉死開關 `_SMART_BRAKE_ON`、
清掉區塊 6 註解的死映射、`_RECORD_WRITER_MAP` 的 5 條死行、`ARCUS_ARCHITECTURE`
與 `FUNCTION_INDEX` 裡的假描述與假封存路徑（`_archived_c3c4/`，該資料夾根本不存在）。
所有活函式逐字保留，1714→1598 行；`py_compile` 通過、server.py 需要的 17 名稱全在、
服務重啟後 `/arcus/` 回 200、`NRestarts=0`。

**教訓**
掏空一個功能群時，必須同步清掉它在三處的殘留：①標頭「幾支合體」宣稱、
②每輪注入系統提示的字串（`_RECORD_WRITER_MAP`／`ARCUS_ARCHITECTURE`）、③`FUNCTION_INDEX`。
否則殘留會每輪對執行中的 agent 謊報不存在的能力，比單純的死碼更有害。
「宣稱數量」與「實際區塊數量」要在同一次改動裡對齊。

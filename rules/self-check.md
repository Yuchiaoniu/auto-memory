# 自檢與驗證規則（最高優先，嚴格執行，無例外）

## 核心原則

**可自行確認的事項，一律自行查驗，禁止詢問使用者。**
目錄是否存在、nginx 是否已設定、服務是否在跑、port 是否被占用——一律直接查驗，不問使用者。

---

## 搜尋必須全面，不得只靠單一路徑

### 反例（錯誤）

用 `which claude` 找不到就斷定「未安裝」。

`which` 只搜尋當前 shell 的 PATH。透過非互動式 SSH 執行時，PATH 是最小預設值，
不含使用者自訂的 `~/.local/bin`、`~/.npm-global/bin`、nvm/pyenv 等版本管理工具路徑。
結果是：工具明明存在、服務明明在跑，卻被誤判為「不存在」而重複安裝。

（2026-06-28 實際發生：GCP 上 Claude CLI 已在 tmux 裡執行 85 小時，ttyd 服務正常運行，
但 `which claude` 在非互動 SSH 下回傳 not found，導致執行了不必要的 npm install。）

### 正確做法：三層查驗，依序執行

**第一層：有沒有在跑？**（最快，不依賴 PATH 或磁碟位置）
```bash
ps aux | grep -v grep | grep <名稱>
```

**第二層：有沒有被 systemd 管理？**（找受管理的服務）
```bash
systemctl list-units | grep -i <名稱>
```

**第三層：磁碟上有沒有執行檔？**（不依賴 PATH）
```bash
find /usr /home /opt /snap /usr/local -name <名稱> -type f 2>/dev/null
```

**補充（第四層）：PATH 找不找得到？**（最後才用，且用 login shell）
```bash
bash -lc "which <名稱>"
```
`-l` 強制載入 `.profile` 和 `.bashrc`，比非互動式 SSH 的 PATH 更完整。

### 規則

- 查驗「工具是否已安裝 / 服務是否存在」時，必須至少跑第一層＋第二層，不得只靠 `which`。
- 任何「找不到」的結論，必須三層都找不到才成立。
- 在 GCP VM 上執行查驗，一律用 login shell（`bash -lc`）或明確三層查驗，禁止只靠非互動式 SSH 的 `which`。

## 自動迭代優化的自檢流程（每次迭代前強制執行）

**系統核心目標（所有六個問題都必須圍繞這句話）：**
辨識使用者對每個科目、每個章節、每個觀念的掌握強弱——
強的觀念停止推送，弱的觀念深度分析原因（而非只記錄答對答錯次數）。
系統應追蹤使用者接觸過哪些觀念、各觀念的理解深度，而非總正確率。

自動迭代優化時，每一輪執行前必須先進入探索模式（explorer），
探索完畢後才能自行判斷最優解，不得死板執行上一輪的邏輯：

每輪迭代前，必須依序回答以下六個問題，回答完畢後才能判斷最優解並執行：

**Q1. 有沒有什麼值得優化的地方？**
上一輪的行為是否產生預期外的結果？資料分布是否出現偏斜、死鎖或覆蓋不全？邏輯假設是否仍然成立？

**Q2. 系統哪些功能值得保留下來？未來會很有用處**
哪些機制現在正常運作，而且未來規模更大時仍然適用？哪些設計決策已驗證有效？不要因為「想優化」而破壞這些部分。

**Q3. 有沒有什麼沒有注意到的問題？**
是否有某個狀態在持續惡化而沒被修正？是否有本應收斂但沒有收斂的指標？任何「早就該發現但一直忽略」的警訊？

**Q4. 有沒有可以降低使用者認知負荷的可能？**
推送的內容是否過量、過密、過雜？有沒有辦法用更少的資訊傳達更清楚的訊息？

**Q5. 這樣修改後有沒有可能出現什麼問題？**
準備做的改動，會不會讓原本正常運作的部分出問題？有沒有邊緣情況會在修改後才冒出來？改完之後需要驗證哪些東西確認沒有壞掉？

**Q6. 有沒有為各項機制做評分？**
針對想要新增、修改或保留的每個機制，給出 0–10 的評分，說明理由。評分必須有比較基準（和上一輪比、和理想狀態比）。

**探索完成後的行動原則：**
- 自行判斷最優解，不得死板複製上一輪的行為
- 遇到兩難時，選擇**最全面、最簡單、最不需要外界干涉**的路線
- 執行前先小規模測試，確認可行後才採用
- 若測試發現問題，重回 Q1，不得強行執行

**早期預警機制（每輪必查）：**
- 迭代 3 輪後，確認所有預期覆蓋的對象都至少被執行過一次
- 若發現覆蓋缺口，立刻觸發警告並修正邏輯
- 不得等到 N 輪後才回頭分析，問題應在 3 輪內被偵測到

---

## 探索模式提案前：先做無變動簡易測試

- 只要處於探索模式，對任何提出的建議，必須先執行一次不修改任何檔案的簡易測試，確認方案可行，才能向使用者回報
- 測試方式：讀程式碼確認邏輯、跑 grep 確認 API 存在、執行診斷指令確認環境，不修改任何檔案
- 測試通過才能說「根本原因是 X，建議修法是 Y」；未測試的方案不得回報

---

## 動手前：先測試假說

- 提出假說後，先執行診斷測試確認假說成立，才動手修改程式碼
- 禁止在假說未經測試的前提下直接修改程式碼

---

## 確認假說後：等待使用者授權，才能動手

- 假說測試完成後，必須先回報「根本原因是 X，建議修法是 Y」
- 等使用者明確說「好」「可以」「實作」「請實作」等授權語，才能開始寫入任何檔案

---

## 實作後：執行驗證，禁止目測代替執行

每次用 Write 或 Edit 寫入程式碼後，必須執行測試腳本驗證邏輯正確、無殘留代碼。
多檔任務可等所有檔案寫完後統一跑測試，但不可跳過。
**禁止在實作後測試通過前向使用者說「已完成」。**

---

## VM 部署後：從外部 IP 發出請求驗證

每次部署或更新 VM 服務後，必須從外部 IP 或公開 URL 實際發出請求，確認回傳預期狀態碼。
**嚴禁只做 `curl localhost`**——localhost 通過不代表外部可達。
部署含 REST API 的服務後，需列出測試矩陣（GET/POST/OPTIONS/未授權），逐一驗證全部通過。
